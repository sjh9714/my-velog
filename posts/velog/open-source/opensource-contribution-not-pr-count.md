# [오픈소스 기여] 오픈소스 기여는 PR 개수 싸움이 아니다

## 들어가며

오픈소스 기여를 시작하면서 처음에는 자연스럽게 PR의 개수에 눈이 갔다.

몇 개를 올렸는지, 몇 개가 merge됐는지, 얼마나 큰 프로젝트에 기여했는지 같은 숫자는 눈에 잘 보인다. 하지만 실제로 PR을 올리고 리뷰를 기다리고 수정 요청을 반영하다 보니, 숫자보다 더 중요한 것이 따로 있다는 생각이 들었다.

오픈소스 기여에서 중요한 것은 PR을 많이 만드는 것이 아니라, 유지보수자가 믿고 검토할 수 있는 증거를 남기는 일이었다.

## 개인 프로젝트와 오픈소스 기여는 보여주는 역량이 다르다

개인 프로젝트는 내가 문제를 정의하고, 구조를 설계하고, 끝까지 구현하는 경험을 보여주기 좋다.

반면 오픈소스 기여는 조금 다르다. 이미 존재하는 코드베이스 안에서 출발해야 한다. 내가 만든 설계가 아니라 다른 사람이 오랫동안 유지해 온 구조를 읽어야 하고, 프로젝트의 테스트 방식과 코드 스타일을 따라야 한다.

그래서 오픈소스 PR은 이런 역량을 보여주기 좋다고 느꼈다.

- 낯선 코드베이스를 빠르게 읽는 능력
- 이슈를 로컬에서 재현하는 능력
- 실패하는 상황을 테스트로 고정하는 능력
- 문제 해결과 무관한 변경을 섞지 않는 능력
- maintainer가 리뷰하기 쉬운 형태로 설명하는 능력

이건 단순히 "코드를 고쳤다"보다 더 실무에 가까운 경험이었다.

## 내가 생각하는 좋은 PR의 기준

좋은 PR은 크고 화려한 PR이 아니라, 검토하기 쉬운 PR에 가깝다.

내가 오픈소스 기여를 하면서 정한 기준은 대략 이렇다.

1. 문제가 현재 코드에서 재현되는가
2. 같은 문제를 해결하는 열린 PR이 이미 없는가
3. 실패 조건을 작은 테스트로 분리할 수 있는가
4. 수정 범위가 한 가지 행동 변화로 끝나는가
5. 검증한 명령어와 검증하지 못한 부분을 설명할 수 있는가

특히 세 번째와 네 번째가 중요했다.

버그를 고치는 것만으로는 부족했다. 같은 문제가 다시 생겼을 때 테스트가 잡아낼 수 있어야 했다. 그리고 테스트를 추가하다 보면 자연스럽게 수정 범위도 줄어든다. 실패하는 상황이 명확해지면, 그 실패를 통과시키기 위한 최소 변경만 남기기 쉬워지기 때문이다.

## PR 개수보다 중요한 것들

오픈소스 기여를 포트폴리오로 정리할 때도 PR 개수만 앞세우면 설득력이 약하다.

더 중요한 것은 각 PR에서 어떤 판단을 했는지다.

- 왜 이 이슈를 골랐는가
- 어떻게 재현했는가
- 원인을 어디까지 좁혔는가
- 어떤 테스트로 회귀를 막았는가
- 왜 이 정도 diff가 충분하다고 판단했는가
- 리뷰 피드백을 어떻게 반영했는가

이 질문에 답할 수 있어야 PR이 경험으로 남는다.

GitHub는 결과를 보여준다. PR 링크, issue 링크, diff, test log가 증거가 된다.

Velog에는 그 과정에서 어떤 생각을 했는지 적을 수 있다. 왜 이 접근을 골랐고, 어디까지 검증했고, 어떤 부분은 의도적으로 하지 않았는지를 정리할 수 있다.

포트폴리오에는 그 둘을 짧게 요약하면 된다.

```text
GitHub는 증거
Velog는 사고 과정
포트폴리오는 요약
```

이 세 가지가 연결될 때 오픈소스 기여가 단순한 기록이 아니라 설득력 있는 경험이 된다고 생각한다.

## AI 도구를 사용할 때의 기준

오픈소스 기여 과정에서 AI 코딩 도구를 활용하기도 했다.

다만 글에서 강조하고 싶은 것은 AI가 아니라 검증이다. 코드 탐색과 수정안 초안 작성에는 AI 도움을 받을 수 있지만, 최종적으로 PR에 들어가는 변경은 내가 이해하고 설명할 수 있어야 한다.

내 기준은 이렇다.

- 저장소의 AI contribution policy를 먼저 확인한다.
- AI가 제안한 수정은 그대로 믿지 않는다.
- 재현, 테스트, diff 검토는 직접 확인한다.
- 검증하지 못한 부분은 PR 본문에 숨기지 않는다.

그래서 앞으로의 글에서도 AI를 전면에 세우기보다는, 어떤 문제를 어떻게 재현하고 어떤 테스트로 확인했는지를 중심으로 정리하려고 한다.

## 앞으로 쓸 글들

이 시리즈에서는 오픈소스 기여를 다음 순서로 정리해보려고 한다.

먼저 좋은 이슈를 고르는 기준을 정리하고, 그 다음에는 버그를 재현하는 방법, 회귀 테스트를 작성하는 방식, 작은 diff를 유지하는 습관을 다룰 예정이다.

이후에는 실제 merged PR 사례를 하나씩 꺼내볼 생각이다.

- Python 패키징 환경에서 app script shebang isolation을 고친 PR
- Go CLI 라이브러리의 struct tag parser edge case를 고친 PR
- lint rule의 optional chaining edge case를 fixture로 고정한 PR
- retry queue에 stale identity가 남는 문제를 고친 PR
- preview cache directory 권한을 제한한 PR

각 글에서는 단순히 "무엇을 고쳤다"보다, "왜 그렇게 고쳤고 어떻게 검증했는가"를 중심에 두려고 한다.

## 마무리

오픈소스 기여는 PR 개수 싸움이 아니었다.

내가 느낀 좋은 기여는 작은 문제를 정확히 재현하고, 유지보수자가 이해할 수 있는 방식으로 수정하고, 같은 문제가 다시 생기지 않도록 테스트를 남기는 일이었다.

앞으로 이 시리즈에서는 그 과정을 하나씩 정리해보려고 한다.
