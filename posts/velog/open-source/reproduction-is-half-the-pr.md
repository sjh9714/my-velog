# [오픈소스 기여] 버그 재현이 PR의 절반이다

## 들어가며

오픈소스 PR을 준비하면서 가장 먼저 하려고 한 일은 코드를 고치는 것이 아니었다.

먼저 실패를 내 손으로 확인하려고 했다. 이슈에 적힌 증상이 지금도 재현되는지, 최신 브랜치에서는 이미 고쳐진 것은 아닌지, 같은 문제를 해결하는 PR이 열려 있지는 않은지 확인하는 과정이 필요했다.

재현하지 못한 버그를 고치는 것은 생각보다 위험하다. 내가 고친 것이 실제 문제인지, 비슷해 보이는 다른 증상인지 구분하기 어렵기 때문이다. 운 좋게 테스트가 통과하더라도, 실제 사용자가 겪은 문제와는 다른 것을 고쳤을 수 있다.

그래서 버그 재현은 PR의 절반에 가깝다고 느꼈다.

## 재현 기록에 남길 것

재현은 단순히 "안 된다"를 확인하는 일이 아니었다.

나중에 PR 본문이나 테스트로 옮길 수 있을 만큼 구체적으로 남겨야 했다. 최소한 다음 항목은 분리해서 기록하려고 했다.

- 실행한 환경
- 사용한 runtime 또는 package manager
- 실행한 명령어
- 기대한 결과
- 실제 결과
- 에러 메시지나 로그
- 최신 main/default branch에서도 같은지 여부

이렇게 적어두면 문제를 다시 볼 때 훨씬 덜 흔들린다.

예를 들어 CLI 버그라면 명령어와 출력이 있어야 한다. parser나 lint rule 버그라면 최소 입력 코드가 있어야 한다. 라이브러리 버그라면 함수 호출과 반환값이 있어야 한다. 이 정도로 작아져야 테스트 위치도 보이기 시작한다.

재현 기록은 PR 본문의 재료가 되기도 한다.

```text
Before: 이 입력에서 기존 동작은 이렇게 실패했다.
After: 같은 입력에서 기대한 결과가 나온다.
Testing: 이 실패 조건을 고정한 테스트와 관련 테스트를 실행했다.
```

이 흐름이 만들어지면 수정 설명이 훨씬 쉬워진다.

## 왜 재현이 수정 범위를 줄이는가

재현은 문제를 크게 만드는 일이 아니라 작게 만드는 일에 가깝다.

이슈를 처음 보면 원인이 넓어 보일 때가 많다. CLI 전체가 잘못된 것 같고, parser 전체를 다시 봐야 할 것 같고, 설정 로딩 전체가 의심될 때도 있다. 하지만 실제로 실패 입력을 줄여보면 문제가 특정 함수, 특정 조건, 특정 fixture로 좁혀지는 경우가 많았다.

재현 과정에서 확인할 수 있는 것도 많다.

첫째, 오래된 이슈인지 확인할 수 있다. 이슈가 열려 있어도 최신 main에서는 이미 고쳐졌을 수 있다.

둘째, 중복 PR을 피할 수 있다. 같은 실패 입력을 검색하면 이미 누군가 같은 문제를 고치고 있는지 보일 때가 있다.

셋째, 테스트를 어디에 추가할지 감이 생긴다. 실패가 CLI 출력 문제인지, 내부 함수 결과 문제인지, parser fixture 문제인지가 분명해지면 테스트 범위도 작아진다.

수정 범위가 작아지는 이유도 여기에 있다. 실패가 작아지면 고칠 코드도 작아진다.

## 선택한 사례: kong mixed bare and kong tags

재현의 중요성을 잘 보여준 사례 중 하나가 [alecthomas/kong#602](https://github.com/alecthomas/kong/pull/602)였다.

이 PR의 제목은 `Fix mixed bare and kong tags`이고, 관련 이슈는 [alecthomas/kong#558](https://github.com/alecthomas/kong/issues/558)이었다.

문제는 Go struct tag 조합에서 나왔다. `required:""`나 `xor:"..."` 같은 bare tag와 `kong:"predictor=..."` tag가 같은 필드에 같이 있을 때, bare tag 쪽이 무시되는 상황이었다. 이슈 설명에 따르면 README에서는 두 tag 형식이 같이 쓰일 수 있다고 되어 있었기 때문에, 기대 동작과 실제 동작이 어긋난 상태였다.

이 문제는 재현 입력을 작게 만들기 좋았다.

큰 CLI 예제를 통째로 가져올 필요가 없었다. embedded struct 하나에 bare `required`/`xor` tag와 `kong:"predictor=..."` tag를 함께 넣으면 실패 조건을 고정할 수 있었다. 그래서 PR에서는 `tag.go`와 `tag_test.go`만 수정했다.

핵심 변경은 bare field tag와 `kong` field tag를 둘 중 하나만 선택하지 않고 함께 해석하도록 하는 것이었다. 동시에 `kong:"-"` ignore 동작과 기존 bare-only, kong-only tag 동작은 유지했다.

이 사례에서 중요한 것은 "tag parser를 전반적으로 개선했다"가 아니었다.

실패 입력이 작았기 때문에 PR도 작아질 수 있었다. 어떤 조합에서 실패하는지, 기대 동작이 무엇인지, 기존 동작 중 무엇을 유지해야 하는지가 비교적 선명했다.

테스트 명령도 좁게 시작할 수 있었다.

```bash
go test ./... -run TestBareAndKongTagsCoexist
go test ./...
```

첫 번째 명령은 새 regression test를 직접 확인하는 명령이고, 두 번째 명령은 기존 전체 테스트 흐름이 깨지지 않았는지 보는 명령이었다. 이런 형태의 재현은 PR 본문에도 그대로 설명하기 좋다.

## 짧은 보조 사례: xq boolean result

CLI에서는 재현이 더 직관적일 때도 있다.

[sibprogrammer/xq#196](https://github.com/sibprogrammer/xq/pull/196)은 그런 예시였다. 이 PR의 제목은 `Fix XPath boolean result output`이었다.

문제는 XPath boolean query 결과 출력이었다. `xq -x 'boolean(//root)'`가 `true`를 출력해야 하는데, 기존에는 `unknown type error: true`처럼 boolean 결과를 알 수 없는 타입으로 처리하는 문제가 있었다.

이런 경우 재현은 명령어와 출력 차이로 정리할 수 있다.

```text
Command: xq -x 'boolean(//root)'
Expected: true
Actual: unknown type error: true
```

수정도 좁게 유지할 수 있었다. PR에서는 `internal/utils/utils.go`와 `internal/utils/utils_test.go`만 바뀌었고, boolean 값을 출력하도록 `XPathQuery` 쪽 처리를 보강했다. 기존 number, string, node iterator 출력 처리는 유지했다.

테스트도 재현 입력을 그대로 따라갔다.

```bash
go test ./internal/utils -run TestXPathQueryBoolean
go test ./internal/utils
go test ./...
```

이 사례는 재현이 PR 설명을 거의 만들어주는 경우였다. 명령어, 기대 출력, 실제 에러가 분리되어 있으니 무엇을 고쳤는지도 분명해졌다.

## 재현이 어려운 경우

물론 모든 이슈가 이렇게 깔끔하게 재현되지는 않는다.

운영체제 차이, 특정 브라우저 버전, 외부 API, 유료 서비스, private credential이 필요한 경우도 있다. 이런 경우에는 억지로 완벽한 재현을 만들었다고 말하지 않는 것이 더 중요했다.

대신 가장 가까운 검증 지점을 찾을 수 있다.

- CLI 전체 재현이 어렵다면 내부 함수 unit test로 줄인다.
- 브라우저 전체 재현이 어렵다면 DOM fixture나 integration test로 줄인다.
- 운영 환경 재현이 어렵다면 실패 조건을 흉내 내는 synthetic test를 만든다.
- 그래도 부족하면 PR 본문에 검증하지 못한 부분을 명시한다.

검증하지 못한 부분은 PR 본문에 숨기지 않고 명시하는 편이 낫다.

오픈소스 PR에서 중요한 것은 모든 환경을 완벽히 재현했다는 척이 아니다. 어디까지 직접 확인했고, 어디부터는 한계가 있었는지 솔직하게 구분하는 것이다.

## 재현은 리뷰어를 위한 배려이기도 하다

재현 기록은 나만을 위한 것이 아니었다.

리뷰어는 PR을 볼 때 전체 맥락을 다시 조사하고 싶지 않을 수 있다. 이슈를 열고, 코드를 읽고, 테스트를 찾아서, 실패를 다시 상상해야 한다면 리뷰 비용이 커진다.

반대로 PR 본문에 재현 흐름이 있으면 리뷰어가 빠르게 판단할 수 있다.

- 이 PR이 어떤 실패를 다루는지
- 그 실패가 현재 코드에서 어떤 입력으로 발생하는지
- 새 테스트가 그 실패를 정말 고정하는지
- 수정 범위가 그 실패에 비해 과하지 않은지

재현이 잘 되어 있으면 PR이 작아 보인다. 코드 줄 수가 적어서가 아니라, 문제와 해결이 같은 선 위에 놓이기 때문이다.

## 마무리

버그를 고치기 전에 실패를 직접 확인하는 과정은 시간이 조금 걸린다.

하지만 그 시간은 대부분 PR 뒤쪽에서 돌아온다. 테스트를 어디에 둘지 빨리 정할 수 있고, 수정 범위를 줄일 수 있고, PR 본문도 더 쉽게 쓸 수 있다. 무엇보다 내가 고친 것이 실제 문제라는 확신을 가질 수 있다.

다음 글에서는 이 재현을 어떻게 regression test로 옮기는지 정리해보려고 한다. 실패하는 테스트를 먼저 만들면 왜 diff가 작아지는지, 그리고 fixture나 snapshot을 고를 때 무엇을 기준으로 삼았는지 이야기해볼 생각이다.
