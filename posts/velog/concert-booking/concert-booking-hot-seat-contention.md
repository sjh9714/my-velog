# [콘서트 예매 시스템] 같은 좌석을 100명이 동시에 잡으면 어떻게 될까

## 이 글의 위치

콘서트 예매 시스템 시리즈는 `정합성 문제 발견 -> 동시성 제어 -> 이벤트 유실 방지 -> 성능/운영 경계` 순서로 읽으면 흐름이 가장 자연스럽다.

1. 좌석은 왜 두 번 팔리면 안 되는가 - 이전 흐름
2. 같은 좌석을 100명이 동시에 잡으면 어떻게 될까 - 현재 글
3. 비관적 락, 낙관적 락, Redis 분산 락을 비교해보았다 - 다음 흐름
4. 대기열 토큰은 왜 예매 트랜잭션 앞에서 검증해야 할까 - 다음 흐름
5. 결제, 만료, 중복 요청이 동시에 오면 상태는 어떻게 지킬까 - 다음 흐름
6. Outbox와 Kafka로 이벤트 유실 구간 줄이기 - 다음 흐름
7. Redis stock은 왜 최종 기준 데이터가 아니어야 할까 - 다음 흐름
8. k6 성능 결과를 어디까지 주장할 수 있을까 - 다음 흐름
9. 운영자가 볼 수 있는 지표와 알림 만들기 - 다음 흐름
10. 면접에서 이 프로젝트를 어떻게 설명할까 - 다음 흐름

## 들어가며

예매 시스템에서 가장 먼저 떠올린 위험은 같은 좌석이 두 번 팔리는 상황이었다.

좌석 A1이 비어 있다. 사용자 1이 A1을 조회한다. 거의 동시에 사용자 2도 A1을 조회한다. 둘 다 "비어 있다"고 판단한다. 둘 다 예약을 생성한다.

코드만 순서대로 읽으면 이런 일이 잘 보이지 않는다.

하지만 요청은 한 줄씩 얌전히 들어오지 않는다. 인기 공연에서는 같은 좌석에 여러 사용자가 동시에 접근할 수 있고, 서버는 그 요청을 거의 동시에 처리한다.

이 글은 `concert-booking`에서 동일 좌석 경합을 어떻게 바라보고, 어떤 방식으로 overselling을 막았는지 정리한 글이다.

Repository는 [sjh9714/concert-booking](https://github.com/sjh9714/concert-booking)이다.

## 예매 시스템에서 가장 위험한 상황

좌석 예매에서 가장 치명적인 오류는 한 좌석이 여러 명에게 팔리는 것이다.

사용자 입장에서는 결제까지 했는데 나중에 좌석이 없다고 말하는 상황이 된다. 운영자 입장에서는 수동 보상과 CS가 필요하다. 시스템 입장에서는 데이터 정합성이 깨진 것이다.

예매 도메인에서 좌석은 공유 자원이다.

여러 요청이 동시에 같은 자원을 읽고 수정하려고 할 때 race condition이 생긴다.

단순한 구현은 보통 이런 흐름을 갖는다.

```text
1. 좌석 조회
2. 좌석이 AVAILABLE인지 확인
3. 좌석 상태를 HELD로 변경
4. 예약 생성
```

문제는 1번과 3번 사이에 다른 요청이 끼어들 수 있다는 점이다.

두 요청이 거의 동시에 1번을 실행하면 둘 다 AVAILABLE을 볼 수 있다. 그리고 둘 다 3번으로 넘어가면 같은 좌석을 두 번 잡는 결과가 생긴다.

그래서 예매 시스템에서는 "조회 후 변경"이 아니라 "동시에 조회하고 변경해도 하나만 성공"하는 구조가 필요하다.

## Race Condition을 코드 밖에서 보기

race condition은 코드 한 줄이 틀려서만 생기지 않는다.

개별 요청만 보면 로직이 맞아도, 여러 요청이 동시에 실행되면 결과가 틀릴 수 있다.

예를 들어 다음 로직은 혼자 실행될 때는 자연스럽다.

```text
if seat.status == AVAILABLE:
    seat.status = HELD
    create reservation
```

하지만 동시에 실행되면 이야기가 달라진다.

```text
T1: A1 is AVAILABLE
T2: A1 is AVAILABLE
T1: A1 -> HELD
T2: A1 -> HELD
T1: create reservation
T2: create reservation
```

이런 상황에서는 "코드가 AVAILABLE을 확인했다"는 사실만으로 충분하지 않다.

확인과 변경 사이를 어떻게 보호할지가 중요하다.

## 이 프로젝트에서 비교한 세 전략

`concert-booking`은 세 가지 reservation strategy를 같은 API 계약에서 비교했다.

- Pessimistic Lock
- Optimistic Lock
- Redis Distributed Lock

목표는 세 전략을 모두 운영 추천안으로 만들기보다, 같은 문제를 서로 다른 방식으로 풀 때 어떤 차이가 생기는지 확인하는 것이었다.

비관적 락은 DB row를 잠그고 순서대로 처리한다. 다른 요청이 같은 row를 수정하려면 기다려야 한다.

낙관적 락은 일단 수정하고 commit 시점에 version 충돌을 확인한다. 충돌이 생기면 재시도하거나 실패시킨다.

Redis 분산 락은 Redis lock과 stock pre-check를 함께 사용한다. 이미 소진된 요청은 DB transaction에 들어가기 전에 빠르게 실패시킬 수 있다.

세 방식은 다르지만, 동일 좌석 경합에서 반드시 지켜야 하는 결과는 하나였다.

한 좌석에는 하나의 성공만 있어야 한다.

## k6로 동일 좌석 경합을 재현하기

동일 좌석 경합은 k6 Scenario A로 확인했다.

조건은 단순하다.

```text
100 VU
동일 좌석 1개
각 VU가 예매 요청 1회
```

즉 100명이 같은 좌석 하나를 동시에 잡으려고 하는 상황이다.

`docs/PERF_RESULT.md`에는 결과가 이렇게 정리되어 있다.

| 메트릭 | 비관적 락 | 낙관적 락 | Redis 분산 락 |
| --- | ---: | ---: | ---: |
| 성공 수 | 1 | 1 | 1 |
| 실패 수 | 99 | 99 | 99 |
| overselling | 0건 | 0건 | 0건 |

이 결과에서 가장 중요한 수치는 p95가 아니었다.

overselling 0건이다.

동일 좌석 1개에 대해 세 전략 모두 하나의 성공만 허용했다. 실패 99건은 시스템이 나쁜 것이 아니라, 이미 잡힌 좌석을 다시 잡으려는 요청을 거절한 결과다.

이 수치는 로컬 Docker 단일 실행 기준이다. 운영 환경 성능이나 capacity를 주장하기 위한 수치가 아니다. 이 글에서는 "동일 좌석 경합에서 정합성 조건을 지켰다"는 근거로만 사용한다.

## Testcontainers로 고정한 것

k6는 실제 HTTP 흐름에 가까운 부하 시나리오를 보여준다.

하지만 테스트 코드에서는 더 작은 단위로 같은 조건을 고정해야 한다.

이 프로젝트에서는 동일 좌석 overselling 방지를 다음 테스트들이 지지한다.

- `ConcurrencyIntegrationTest`
- `DistributedLockConcurrencyTest`
- `OptimisticLockConcurrencyTest`

테스트의 목적은 단순했다.

여러 요청이 같은 좌석을 예약하려고 해도 최종적으로 하나의 예약만 성공해야 한다.

이런 테스트가 있으면 전략을 바꾸거나 내부 구현을 수정할 때도 기본 불변식을 확인할 수 있다.

## 실패는 정상 동작일 수 있다

동시성 테스트를 볼 때 헷갈릴 수 있는 부분이 있다.

실패가 많으면 시스템이 나쁜 것처럼 보일 수 있다.

하지만 동일 좌석 1개를 100명이 동시에 잡는 상황에서는 99명이 실패해야 정상이다. 모두 성공하면 그게 문제다.

예매 시스템에서 실패 응답은 정합성을 지키기 위한 결과일 수 있다.

중요한 것은 실패가 어떤 이유로 발생했는지다.

- 좌석이 이미 HELD 또는 RESERVED라서 실패했는가
- 대기열 token이 없어서 실패했는가
- idempotency conflict라서 실패했는가
- 결제/만료 race에서 이미 상태가 바뀌어서 실패했는가

실패를 모두 같은 실패로 보지 않고, 도메인 의미별로 분리해야 한다.

## 성능보다 먼저 정합성

동일 좌석 경합에서 p50, p90, p95 같은 latency도 볼 수 있다.

하지만 이 글에서 먼저 보고 싶은 것은 정합성이다.

좌석이 두 번 팔리는 시스템은 아무리 빨라도 위험하다. 반대로 정합성을 지키는 시스템은 그 다음에 성능을 개선할 수 있다.

그래서 이 프로젝트의 첫 번째 기준은 다음 문장이었다.

```text
한 좌석은 한 번만 팔린다.
```

그 다음에야 어떤 전략이 어떤 상황에서 더 적합한지 비교할 수 있다.

## 배운 점

동시성 문제는 로직을 순서대로 읽는 것만으로는 잘 보이지 않았다.

개별 요청에서는 맞는 코드가 동시에 실행되면 틀릴 수 있다. 그래서 같은 좌석에 요청을 몰아넣는 테스트가 필요했다.

또 하나 배운 점은 실패 응답을 부정적으로만 보면 안 된다는 것이다.

이미 선점된 좌석을 거절하는 실패는 정합성을 지키는 정상 동작이다. 예매 시스템에서는 성공 수만큼 실패 이유도 중요하다.

## 포트폴리오 메모

동일 좌석 100 concurrent requests 상황을 k6와 Testcontainers 통합 테스트로 검증해, 세 reservation strategy 모두 success 1, fail 99, overselling 0 조건을 만족하도록 했다. 이 수치는 로컬 Docker 단일 실행 기준이며, 운영 성능이 아니라 좌석 정합성 검증 근거로 사용했다.
