# [하나 금융인재 프로젝트] 학습 기록을 포트폴리오로 재구성한 이유

## 한 줄 요약

하나 청년 금융인재 양성 프로젝트에서 배운 내용을 날짜별 학습 로그로만 남기지 않고, 금융 데이터와 AI 서비스를 이해해가는 포트폴리오 흐름으로 다시 정리했다.

## 문제 정의

처음에는 `day3`, `day4`, `day10`처럼 날짜별 폴더에 배운 내용을 쌓아두는 것만으로 충분하다고 생각했다.

그런데 GitHub 저장소를 외부 사람이 본다고 생각하면 문제가 있었다.

- 어떤 내용을 배웠는지는 보인다.
- 꾸준히 학습한 흔적도 보인다.
- 하지만 그래서 무엇을 만들 수 있는 사람인지가 바로 드러나지는 않는다.

그래서 이번에는 저장소의 역할을 다시 나누었다.

```text
GitHub = 증거
Velog = 설명과 회고
README = 입구
```

GitHub에는 실제 학습 파일과 코드가 있고, Velog에는 내가 그 내용을 어떻게 이해했는지 적는다. README는 그 둘을 연결하는 첫 화면으로 둔다.

## 실습 코드와 구조

`hana_power_on_study` 저장소에는 Python 기초부터 Pandas, 금융 데이터 구조, Gradio, LangChain, RAG, LangGraph까지 여러 학습 기록이 들어 있다.

폴더 구조는 대략 이런 흐름이다.

```text
step1
  Python, Pandas, 시계열, 신용카드 거래 데이터 실습

day3~day9
  Python 기초, 데이터 처리, 금융 데이터 구조, FDS 기초

day10~day12
  프롬프트 엔지니어링, Gradio, OpenAI, LangChain 챗봇

day13~day16
  PDF 처리, Chroma, RAG, Router, LangGraph Agent
```

이 구조는 학습 순서를 보존하기에는 좋다. 하지만 포트폴리오로는 조금 약하다. 채용자 입장에서는 `day13`이라는 이름보다 “PDF 문서를 검색 가능한 지식베이스로 바꿨다”는 설명이 훨씬 빨리 이해된다.

이번 글에서 새로 만든 코드는 없지만, 기존 실습 폴더를 아래 문서 흐름으로 다시 읽을 수 있게 정리했다.

```text
README.md
  -> docs/portfolio-index.md
  -> docs/learning-log.md
  -> docs/blog-roadmap.md
  -> 기존 day*/step1 실습 폴더
```

즉, 코드는 그대로 두고 읽는 순서를 새로 만든 작업이다.

## 재구성 방향

그래서 README 상단에 다음 메시지를 먼저 배치했다.

> 금융 서비스 문제를 데이터 분석, AI 인터페이스, 백엔드 구조로 풀어내기 위한 학습 기록

이 문장을 기준으로 대표 트랙을 4개로 나눴다.

| Track | 보여주고 싶은 역량 |
| --- | --- |
| Credit Card Transaction EDA | 금융 거래 데이터를 Pandas로 분석하는 기초 |
| Finance Prompt Engineering Lab | 금융 안내 챗봇에서 프롬프트를 요구사항처럼 설계하는 관점 |
| FastAPI + Gradio AI Chat Demo | AI 기능을 사용자 인터페이스로 빠르게 검증하는 흐름 |
| RAG + LangGraph Agent Chatbot | 문서 검색, Tool 호출, Agent 흐름을 나누는 구조 |

핵심은 “수업을 들었다”가 아니라 “배운 내용을 어떤 문제 해결 흐름으로 이해했는가”를 보여주는 것이다.

## 헷갈린 점

가장 헷갈렸던 부분은 “학습 기록”과 “포트폴리오”의 차이였다.

학습 기록은 내가 배운 순서대로 남기면 된다. 하지만 포트폴리오는 읽는 사람이 빠르게 맥락을 잡을 수 있어야 한다.

그래서 날짜별 폴더를 없애기보다, 그 위에 대표 트랙과 설명 문서를 얹는 방식이 더 맞다고 봤다.

## 금융 서비스와 연결

이 저장소의 방향은 단순 AI 챗봇 실습이 아니라 금융 서비스에 필요한 데이터와 AI 흐름을 이해하는 것이다.

금융 서비스에서는 데이터 분석, 사용자 응대, 문서 검색, 자동화 흐름이 따로 떨어져 있지 않다. 거래 데이터를 이해하고, 사용자의 질문을 구조화하고, 내부 문서를 검색하고, 필요한 도구를 호출하는 흐름이 함께 필요하다.

이번 재구성은 그 흐름을 README와 Velog 시리즈로 보이게 만드는 첫 단계다.

## Velog에는 무엇을 쓸 것인가

Velog에는 강의 내용을 그대로 요약하지 않기로 했다.

대신 글마다 이런 구조로 정리하려고 한다.

```text
문제 정의
-> 실습 코드 또는 구조
-> 헷갈렸던 점
-> 금융 서비스와 연결하면?
-> 한계와 개선점
-> GitHub에서 확인할 수 있는 것
```

이렇게 하면 글이 단순 필기에서 끝나지 않고, 내가 어떤 식으로 문제를 이해했는지 보여줄 수 있다.

## 공개할 때 조심할 점

금융과 교육 과정이 섞여 있기 때문에 공개 범위도 중요했다.

공개해도 되는 것은 직접 작성한 코드, README, 회고, 샘플 구조다.

반대로 조심해야 하는 것은 다음과 같다.

- 교육 자료 원문
- 내부 제공 데이터
- 실제 개인정보나 금융 거래 원본
- API key, token, secret
- 실제 금융 조언처럼 보이는 단정적 표현

그래서 `docs/public-data-policy.md`에 공개 기준을 따로 정리했다.

## 한계와 개선점

이번 정리는 1차 정리다.

아직 저장소 안에는 대형 데이터 압축 파일, Chroma DB 산출물, PDF 처리 결과물이 남아 있다. 이번 작업에서는 삭제하지 않고 문서화만 했다. 나중에는 샘플 데이터와 재생성 스크립트 중심으로 바꾸는 편이 더 깔끔할 것 같다.

또한 각 대표 트랙마다 별도 README와 실행 화면을 붙이면 포트폴리오 완성도가 더 올라갈 수 있다.

## 마무리

학습 기록은 쌓는 것만으로도 의미가 있지만, 포트폴리오에서는 읽는 사람이 빠르게 이해할 수 있어야 한다.

이번 정리의 목표는 그 중간 지점을 만드는 것이었다.

이제부터는 GitHub에 남은 코드를 증거로 두고, Velog에서는 내가 왜 그렇게 이해했는지와 어떤 한계가 있었는지를 더 자세히 적어보려고 한다.
