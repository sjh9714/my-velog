## 완주하지 못한 선수: list.remove 시간 초과

처음에는 완주한 사람을 참가자 목록에서 하나씩 지우면 된다고 생각했다.

```python
def solution(participant, completion):
    for i in completion:
        participant.remove(i)
    return participant[0]
```

이 풀이는 작은 입력에서는 맞는다.
그래서 정확성 테스트는 통과했다.

하지만 `list.remove()`는 리스트에서 값을 찾기 위해 앞에서부터 탐색한다.
완주자 수만큼 `remove()`를 반복하면 매번 리스트 탐색이 발생한다.
결국 입력이 커질수록 `O(n^2)`에 가까워져 효율성 테스트에서 시간 초과가 날 수 있다.

이 문제에서 더 중요한 조건은 동명이인이 있을 수 있다는 점이다.
이름이 있는지만 확인하는 `set`으로는 부족하고, 이름별 개수를 세야 한다.

그래서 다음에는 이런 신호를 보면 바로 `Counter`나 `dict`를 떠올리기로 했다.

- 같은 값이 여러 번 나올 수 있다.
- 어떤 값이 몇 번 등장했는지 비교해야 한다.
- 리스트에서 값을 반복해서 찾거나 삭제하고 있다.
- 효율성 테스트가 있는 문제다.

정리하면, 이 문제의 핵심은 “완주하지 못한 한 명 찾기”가 아니라 “두 목록의 이름별 개수 차이 찾기”였다.

## 이상한 문자 만들기: split이 공백을 지운다

처음에는 문자열을 단어 단위로 나누면 쉽게 풀릴 것 같았다.

```python
def solution(s):
    answer = ''
    arr = s.split()
    for i in arr:
        for j in range(len(i)):
            if j % 2:
                answer += i[j].lower()
            else:
                answer += i[j].upper()
        answer += ' '
    return answer.strip()
```

기본 예제처럼 단어 사이 공백이 하나씩만 있으면 잘 동작한다.
하지만 정확성 테스트가 31.3점에서 멈췄다.

원인은 `split()`이었다.

`split()`은 단어를 나누기에는 편하지만, 연속된 공백과 앞뒤 공백 정보를 잃는다.

```python
"hello  world".split()
# ["hello", "world"]
```

이 문제는 단어를 바꾸는 문제처럼 보이지만, 실제로는 원래 문자열의 공백까지 그대로 유지해야 하는 문제였다.
그래서 공백을 새로 만들어 붙이면 안 되고, 입력에 있던 공백을 그대로 결과에 넣어야 했다.

다음에는 이런 신호를 보면 `split()`을 바로 쓰기 전에 한 번 멈추기로 했다.

- 공백이 여러 개 들어올 수 있다.
- 앞뒤 공백도 유지해야 한다.
- “원래 문자열과 같은 위치”가 중요하다.
- 공백 자체가 결과의 일부다.

이 문제의 핵심은 “단어 리스트로 변환하기”가 아니라 “원본 문자열을 보존하면서 문자만 바꾸기”였다.

## 콜라츠 추측: 처음부터 1인 경우

콜라츠 추측은 홀수와 짝수 규칙을 반복하면 되는 문제였다.

처음 코드는 대부분의 테스트를 통과했지만, 테스트 13번만 실패했다.

```python
def solution(num):
    index = 0
    while index < 500:
        if num % 2:
            num = num * 3 + 1
        else:
            num /= 2
        index += 1
        if num == 1:
            return index
    return -1
```

핵심 반례는 `num = 1`이었다.

문제에서 이미 `num`이 1이면 작업을 한 번도 하지 않아도 된다.
따라서 정답은 `0`이어야 한다.

하지만 처음 코드는 반복문에 들어간 뒤에야 `num == 1`을 확인했다.
그래서 `1`을 이미 정답인 상태로 보지 못하고, 홀수 규칙을 적용해 `4`로 바꿔버렸다.

이 문제에서 배운 점은 단순하다.

반복문 문제에서는 반복을 시작하기 전에 이미 끝난 상태인지 먼저 확인해야 한다.

다음에는 이런 신호를 보면 경계값부터 넣어보기로 했다.

- 반복해서 목표 상태에 도달하는 문제다.
- 입력이 이미 목표 상태일 수 있다.
- 최대 반복 횟수가 정해져 있다.
- 테스트 하나만 실패한다.

특히 `num = 1`, 빈 배열, 길이 1 문자열처럼 가장 작은 입력은 먼저 확인해야겠다.

## 1주차에 반복해서 나온 실수

1주차를 돌아보면, 문법 자체보다 “언제 어떤 도구를 써야 하는지”에서 자주 막혔다.

첫 번째는 자료구조 선택이다.

리스트는 순서대로 접근할 때는 편하지만, 특정 값을 계속 찾거나 삭제할 때는 느려질 수 있다.
반대로 `dict`, `set`, `Counter`는 조회나 개수 계산이 필요할 때 훨씬 자연스럽다.

두 번째는 문자열 함수의 부작용이다.

`split()`, `strip()`, `replace()`, `join()`은 모두 편리하지만, 원본의 어떤 정보가 사라지는지 생각해야 한다.
특히 공백을 유지해야 하는 문제에서는 `split()`이 오히려 위험할 수 있었다.

세 번째는 경계값 확인이다.

대부분의 테스트는 통과했는데 하나만 실패한다면, 로직 전체가 틀렸다기보다는 특정 입력을 놓쳤을 가능성이 크다.
`num = 1`, 전부 0, 빈 값, 길이 1, 중복 값 같은 입력을 먼저 확인해야 한다.
